# Py Safe Grading Framework

Testing infrastructure for safely testing student code, with easy access to testing specific functions,
entire script files, performing AST checks, and style checks. Output feedback clearly specifies, what
function return types should be, if they should or should not print output etc, with output and stderr formatting
showing hidden chars such as newlines while also preserving line wrapping to make it easier for students to understand why their output differs.

The primary goal of this project is to make it as easy as possible to setup whatever test is required for an assignment. The existing tooling inside Edstem, and with test libraries such as PyUnit are not suitable.

Issues that this project solves:
- If a student crashes the testing infrastructure code they get 0, even if they passed preceeding test cases
    - Every test is run as a seperate subprocess with a time limit, so the tests are
      independant, and cannot influence the main process unless there is a bug which needs to
      be patched.
- Using input output tests, does not allow for particularly descriptive outputs or type checking
    - Can directly test python objects for types etc
- Custom checks are time consuming to setup
    - A comprehensive set of features are provided
- Hidden/Private testcases are possible to leak if not carefully mitigated during setup
    - This project enforces many constraints to prevent data leakage

Configured for use with Edstem, but could be extended to work with other platforms with minimal effort by taking the output json object from `SafeGrading.run_tests` and adjusting the format as required (or creating a further modified version of this function if more access to the internal test_data is required as the output format is seperate from the internal data).

Developed primarily for COMP10001 at the University of Melbourne, but has also been used for other subjects.

### Documentation
For full documentation see [Documentation](https://comp10001.github.io/py-safe-grading-framework), or to build it locally use:

```bash
python -m venv .venv
source .venv/bin/activate # or equivalent for your OS
pip install -r requirements.txt
pip install -r doc-requirements.txt
python -m sphinx -b html docs/source docs/build/html
python -m http.server 8080 -d docs/build/html
```
Then you can access the documentation on localhost at `http://127.0.0.1:8080`.


### Rough Internal Test Execution Summary
- run testbench python file
- import safe grading
    - Import priority order is changed so student files cannot shadow built ins
    - Redirect stdout to /dev/null so cannot accidentally print and break the output json format
- create `SafeGrading` object -> dev_and_prod_mode_options() called. With `--prod` cli flag all python files imported by testbench are removed from path.
- Optionally cache hidden test files so they are removed from path until a given test is running (specified to reveal when registering a given test), or else the file contents could be leaked by printing the contents through a visible test.
- Now all tests are registered before reaching run_tests
- run_tests() enforces a memory limit below the maximum for the container
- import priority order is reverted (resetting the path variable back to as it was)
- `__pycache__` is removed to prevent data leakage of testcases from imports into testbench (for example if testcase inputs in a seperate file)
- each testcase is run in a seperate subprocess
    - all visible tests are done before any hidden/private tests so that data leakage from a preceeding test is not possible
    - a test runner python file is generated for each testcase and run as a subprocess to produce the outputs needed for each test
    - on load the test runner file removes itself from the path so it does not interfere with the test
    - the test runner imports are loaded with the local path excluded (so that code injection is not possible) before restoring the path so the python code can run the same as it would when run seperately.
    - some patches are optionally applied to student code such as input echoing to stdout (for tty emulation in the output) or recursive call counting etc.
- The main process loads the outputs generated by the subprocess and scores the test, before running the next test
    - This loads data such as stdout, stderr, function return values from a file output by the test runner
    - Can add adjustments at the very end of this stage using the custom verification override function interface
- On completion of all tests extra files are optionally generated such as test report or execution transcript for easy reference
- Output json format is generated. Ed strictly allows output of 200,000 chars or less so feedback messages are iteratively shortened until the json object fits within the limits or a setup error is generated (too many testcases).
- Result JSON object is printed to STDOUT for Ed to load
